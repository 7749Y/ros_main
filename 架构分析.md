# ROS 项目架构分析

## 现状

当前 10 个包分布在 2 个工作空间：

```
src/         → 6 包 (arm_control, ar_tf_arm_5, auto_bkhome(空), auto_charge, nav_goal, romove)
src_task_1/  → 4 包 (auto_bkhome, auto_nav_goal_4, auto_nav_goal_5, Grab)
```

**核心问题**: `navToGoal()`、`set_relmove()`、`arm_move()`、`set_ARtrack()` 等函数在 7 个文件中被重复实现，代码复制粘贴率超 70%。**没有代码复用机制**。

---

## 推荐架构：共享库 + 参数分离 + launch 编排

重构为**一个工作空间 + 三个包**：

```
src/
├── robot_core/              # 包1: 共享库
│   ├── include/robot_core/
│   │   ├── arm_controller.h     # arm_move(), set_pump()
│   │   ├── navigation.h         # navToGoal()
│   │   ├── relocalization.h     # set_ARtrack(), set_relmove()
│   │   └── tf_utils.h           # get_transform()
│   └── src/
│       ├── arm_controller.cpp
│       ├── navigation.cpp
│       ├── relocalization.cpp
│       └── tf_utils.cpp
│
├── robot_apps/              # 包2: 功能节点 (链接 robot_core 库)
│   ├── src/
│   │   ├── nav_goal_node.cpp
│   │   ├── auto_charge_node.cpp
│   │   ├── inbound_operation_node.cpp
│   │   ├── grab_node.cpp
│   │   └── bkhome_node.cpp
│   └── launch/
│       ├── nav_goal.launch
│       ├── auto_charge.launch
│       ├── inbound.launch
│       ├── grab.launch
│       └── bkhome.launch
│
└── robot_params/             # 包3: 配置文件 (纯 YAML)
    └── config/
        ├── nav_goals.yaml     # 所有导航目标点
        ├── arm_poses.yaml     # 所有机械臂位姿
        ├── ar_tags.yaml       # AR 标签 ID 映射
        └── timing.yaml        # 延时/超时参数
```

### 改进点一：共享库消除代码重复

```cpp
// ❌ 当前: 7 份独立实现
if (!navToGoal(0.49384, 1.26898, 0, 1)) ...

// ✅ 重构后: 统一调用共享库
#include <robot_core/navigation.h>
robot_core::navToGoal(client, x, y, z, w);
```

### 改进点二：参数与代码彻底分离

```cpp
// ❌ 当前: 坐标写死在源码中
if (!navToGoal(0.49384, 1.26898, 0, 1)) ...

// ✅ 重构后: 从参数服务器读取
double gx, gy, gz, gw;
nh.getParam("nav_goal/x", gx);
nh.getParam("nav_goal/y", gy);
nh.getParam("nav_goal/z", gz);
nh.getParam("nav_goal/w", gw);
if (!navToGoal(gx, gy, gz, gw)) ...
```

YAML 文件作为单一真相来源：

```yaml
# config/nav_goals.yaml
charge_station:   {x: 0.3, y: 1.75, z: 0.7, w: 0.7}
shelf_a:          {x: 0.49, y: 1.27, z: 0.0, w: 1.0}
home:             {x: 0.003, y: -0.001, z: -0.012, w: 1.0}
```

### 改进点三：launch 文件编排任务

```bash
# 启动单一功能
roslaunch robot_apps grab.launch

# 组合完整流程
roslaunch robot_apps full_workflow.launch
```

`full_workflow.launch` 示例：

```xml
<launch>
  <include file="$(find robot_apps)/launch/bkhome.launch"/>
  <include file="$(find robot_apps)/launch/nav_goal.launch">
    <arg name="goal" value="shelf_a"/>
  </include>
  <include file="$(find robot_apps)/launch/grab.launch"/>
</launch>
```

## 可选补充：聚合入口节点

如想保留"一个可执行文件通过参数选功能"的方式，可额外提供一个聚合节点：

```cpp
// robot_apps/src/commander.cpp
int main(int argc, char** argv) {
    ros::init(argc, argv, "robot_commander");
    std::string mode;
    nh.getParam("~mode", mode);

    if (mode == "grab")     run_grab_sequence(params);
    else if (mode == "charge") run_charge_sequence(params);
    else if (mode == "home")   run_bkhome(params);
    // ...
}
```

```bash
rosrun robot_apps commander _mode:=grab
```

但这应作为 launch 方案的**补充**而非替代。launch 的优势（多节点编排、remap、namespace）是单进程方式做不到的。

---

## 总结

| 当前问题 | 改进方案 |
|---------|---------|
| 7 份重复代码 | `robot_core` 共享库 |
| 坐标/ID/延时写死在源码 | `robot_params` + 参数服务器 |
| 每功能一个包，管理混乱 | 统一工作空间，三包结构 |
| 无编排机制 | launch 文件组合任务 |
